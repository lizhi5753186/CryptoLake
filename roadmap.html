先看清岗位到底要什么

别被 JD 里一长串技术名词吓到。把招聘要求翻译成"能力",你会发现优先级差别很大。下面是加密/互联网数据分析师岗位的真实权重——你的时间要按这个比例分配,而不是平均用力。

🥇 SQL + 业务解读 最高权重

面试 80% 的现场题是 SQL。但真正拉开差距的是:你能不能把一个业务问题(留存下降)翻译成一句查询,再把查询结果翻译回业务结论。这是本路线的核心。

🥈 Python + 统计 高

pandas 做清洗与分析、matplotlib 画图、基础统计与 A/B 测试。不用会算法工程,但要能独立跑通一个分析、说清置信度。

🥉 大数据 + BI 加分

Hive/Spark SQL、数仓分层、调度、以及一个能给业务看的看板(Superset/Tableau/PowerBI/FineBI)。校招阶段是明显的差异化亮点。

一句话心法: 招聘方不缺"会写 SQL 的人",缺"能用数据把业务问题说清楚的人"。所以这套规划里,每写一段代码都要求你补一句"这说明了什么业务结论"——这也是你面试时最能打的东西。
01

主线项目:CryptoLake 交易平台数据仓库

加密交易平台的商业模式很干净:平台靠交易手续费赚钱。于是数据分析师的所有问题都围绕一件事——怎么让更多的人、更频繁地、交易更多的量。你要扮演这个分析师,自己造出一套贴近真实的数据,然后回答这些问题。

业务场景(你的"甲方")

  • 运营问:新用户第 7 天还回来吗?(留存)
  • 市场问:哪个渠道来的用户最能贡献手续费?(渠道 ROI)
  • 产品问:大户(鲸鱼)和散户行为差在哪?(分层)
  • 财务问:手续费收入靠哪些币种/用户支撑?(营收拆解)
  • 风控问:哪些账户交易行为异常?(异常检测)

没有真实数据?自己造 —— 这本身就是能力

用 Python(Faker + numpy)按真实分布造数,比随机数高级得多,面试官会追问:

  • 交易频次用泊松分布,少数鲸鱼贡献大部分量用幂律分布
  • 价格用随机游走生成 K 线,避免"假得离谱"
  • 故意埋入流失用户、异常账户,让后面的分析有东西可挖

数据模型(MySQL 维度建模)

表名类型说明 / 关键字段
dim_user维度用户:注册时间、国家、KYC等级、注册渠道、referrer_id、VIP等级
dim_asset维度币种:BTC/ETH/USDT…、类别(主流/山寨/稳定币)
dim_date维度日期表:年/月/周/是否周末,做时间分析必备
fact_trade事实成交明细:user_id、symbol、side、price、qty、fee(手续费)、ts
fact_order事实挂单:含未成交/撤单,用于挂单成交率分析
fact_deposit_withdraw事实充值/提现:资金流入流出、净入金
fact_login事实登录/活跃日志:算 DAU/MAU、活跃度
fact_kline事实K线 OHLCV:行情数据,做量价关联

✳️ 一句话讲清架构:1 张手续费事实表(fact_trade)是钱的来源,其它表都围绕它。面试时你能画出这张星型模型,就已经赢过一大半候选人。

两段"能上简历"的核心 SQL

下面是 P2 阶段你会亲手写出来的东西。它们同时也是 SQL 面试的高频考点(留存 cohort、RFM)。看不懂没关系,这正是你 3 个月后要能默写的水平。

📈 次日/7日留存 · 用窗口函数算 cohort
-- 每个注册日的用户,在第 N 天是否回来交易
WITH first_day AS (
  SELECT u.user_id, DATE(u.reg_time) AS cohort_day
  FROM dim_user u
),
activity AS (
  SELECT DISTINCT user_id, DATE(ts) AS act_day
  FROM fact_trade
)
SELECT
  f.cohort_day,
  COUNT(DISTINCT f.user_id)                          AS cohort_size,
  COUNT(DISTINCT CASE WHEN DATEDIFF(a.act_day,f.cohort_day)=1
        THEN f.user_id END)                          AS d1_retained,
  COUNT(DISTINCT CASE WHEN DATEDIFF(a.act_day,f.cohort_day)=7
        THEN f.user_id END)                          AS d7_retained
FROM first_day f
LEFT JOIN activity a ON a.user_id = f.user_id
GROUP BY f.cohort_day
ORDER BY f.cohort_day;
💰 RFM 用户分层 · 找出高价值用户
-- R=最近一次交易, F=交易次数, M=贡献手续费; 各分 5 档打分
WITH rfm AS (
  SELECT user_id,
    DATEDIFF(CURDATE(), MAX(DATE(ts))) AS recency,
    COUNT(*)                              AS frequency,
    SUM(fee)                              AS monetary
  FROM fact_trade
  GROUP BY user_id
)
SELECT user_id, recency, frequency, monetary,
  NTILE(5) OVER (ORDER BY recency DESC)  AS r_score,
  NTILE(5) OVER (ORDER BY frequency)      AS f_score,
  NTILE(5) OVER (ORDER BY monetary)       AS m_score
FROM rfm;
-- 之后:R低F高M高 = 忠实鲸鱼;R高F低 = 流失预警,该发召回券
